# 01 — 專案總覽

## 1.1 目的

提供物理治療師在「評估個案」時可用的視覺化工具，達成三個目標：

1. **可視覺化**：以 3D 人體模型呈現肌肉、神經與關節活動，協助治療師理解與定位問題。
2. **醫病溝通**：在個案面前以模型與圖示說明問題位置與動作受限，降低口頭解釋的門檻。
3. **紀錄個案**：以標準化評估表（SFMA）記錄評估結果與個案資訊，供後續追蹤。

## 1.2 目標使用者

| 角色 | 說明 | 主要需求 |
| --- | --- | --- |
| 物理治療師（主要） | 操作 APP 進行評估與衛教 | 快速輸入評估、即時操作人體模型、查看歷史紀錄 |
| 個案／病患（被動） | 觀看治療師展示的模型與說明 | 直覺、易懂的視覺呈現 |
| 系統管理者（未來） | 管理後端資料與帳號 | 資料同步、權限管理 |

## 1.3 使用情境

- **臨床評估**：治療師依 SFMA 流程逐項評估，於 APP 勾選判讀結果（FN／FP／DN／DP），系統計算總覽。
- **衛教溝通**：治療師旋轉人體模型、開關肌群／神經分層、放大特定關節，向個案說明動作受限與相關肌群。
- **裝置切換**：診間以 PC／平板（橫式）操作；隨身評估以手機（直式）快速記錄。
- **效能適配**：預設使用 3D 精簡版；可手動切換 3D 完整（無損）版，並於 FPS 不足時自動降級。

## 1.4 兩大核心功能

```
物理治療 APP
├── 人體模型（視覺化與溝通）
│   └── 3D 版：可旋轉、可活動（受 ROM 限制）、LOD 精簡／完整
└── 評估表（紀錄與判讀）
    └── SFMA：10 大動作 → FN/FP/DN/DP → 二階 Breakout
```

兩者並非各自獨立。設計上以「**解剖實體（肌肉／神經／關節）**」作為共同資料基礎，使評估結果可標註回人體模型對應部位（見 [06_data_model.md](06_data_model.md) 的 `bodyAnnotation`），達成「評估—視覺化」的連動。

## 1.5 範圍（開發期 vs 未來擴充）

> 此處為「設計範圍界定」，非開發時程。

| 面向 | 開發期 | 未來擴充 |
| --- | --- | --- |
| 個案資料來源 | 手動輸入 | 自動匯入／與院所系統介接 |
| 資料儲存 | **單機本地儲存（IndexedDB）+ JSON 匯出／匯入** | **資料上傳到 PostgreSQL**（後端 API、同步） |
| 執行環境 | 瀏覽器 Web／PWA；3D 模型資產為雲端服務（CDN） | App Store 上架（Capacitor 包裝同一程式碼） |
| 評估方法 | SFMA 頂層 10 大動作 + 二階 Breakout 紀錄 | 其他量表（FMS、特殊測試）外掛 |
| 人體模型 | 3D 簡化／完整（肌肉、神經、關節、筋膜） | 動作錄製、客製化標註範本 |
| 帳號與多人 | 單機、無帳號 | 治療師帳號、權限、雲端同步 |

## 1.6 核心設計原則

- **資料驅動**：解剖結構、評估題項、判讀標準皆以資料（JSON）定義，UI 由資料生成，便於擴充與多語系。
- **Web 優先**：以瀏覽器開啟，3D 模型資產為雲端服務（CDN 串流）；個案資料**開發期單機本地儲存（IndexedDB）並可匯出／匯入**，未來上傳 PostgreSQL（見 [02_architecture.md](02_architecture.md)）。
- **效能分級**：以 3D LOD 切換適配不同運算能力裝置，並依裝置能力下載對應資產（兼顧效能與頻寬），預設自動偵測、允許手動覆寫。
- **跨平台一致**：同一套 Web／PWA 程式碼經瀏覽器涵蓋 iOS／iPadOS／Android／Windows，介面隨方向與裝置尺寸響應式調整。
- **資料保護**：最小蒐集；開發期資料不離開裝置（除主動匯出），未來上雲時全程加密與存取隔離，符合個資法（見 [08_security_privacy.md](08_security_privacy.md)）。
